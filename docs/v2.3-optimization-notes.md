# v2.3 优化落地说明

## 背景

v2.3 已经把系统从泛 AI 新闻聚合升级为“按头部大模型产品线组织”的官方信源日报。真实运行后暴露出两个需要继续优化的问题：

1. OpenAI 官方 RSS 能发现 ChatGPT/Codex 相关新闻，但正文页经常返回 403，导致高置信线索容易被丢弃。
2. 部分官方 changelog 页面会把旧锚点、页面标题或英文标题带入候选，影响日报的中文可读性和排序质量。

本轮优化的目标是保持官方源优先，同时提升日报的稳定性和可读性。

## 本轮改动

### 1. 官方 registry 源支持摘要降级

当官方注册源已经从 RSS、JSON 或列表页拿到标题、摘要和链接，但正文页抓取失败时，系统会保留该摘要作为 `excerpt_only` 候选。

适用范围：

- `discovery_source=registry`
- `discovery_source=whitelist`
- 补救阶段显式允许 `allowExcerptOnly`

这样 OpenAI 等站点出现 403 时，日报仍可利用官方 RSS 摘要完成判断，不会直接丢失产品线覆盖。

### 2. OpenAI 新闻源过滤宣传型内容

OpenAI Newsroom 继续作为 ChatGPT/Codex 的官方新闻源，但新增排除规则，降低客户案例、合作宣传和企业案例进入日报的概率。

典型过滤对象：

- 合作伙伴公告
- 客户案例
- “某团队如何使用 Codex”类内容
- 企业采用/宣传型故事

保留对象：

- ChatGPT 产品能力更新
- Codex 产品更新
- 模型能力更新
- 面向用户或开发者的真实功能变化

### 3. 未来日期不再被当作最新内容

部分官方文档锚点可能出现类似 `12-13-23` 的旧日期片段。如果解析器误把它当成未来日期，旧内容会被错误判定为“最新”。

现在评分逻辑会把超过当前时间 24 小时以上的未来日期按低新鲜度处理，避免旧 changelog 条目冲到排序前列。

### 4. 产品线标题统一标准化

当模型返回 `gemini` 这类小写英文产品线标题时，日报会自动映射为标准产品名，例如：

- `gemini` -> `Gemini`
- `openai_codex` -> `Codex`
- `deepseek` -> `DeepSeek`

这能减少日报里的格式噪音，让飞书和 GitHub Pages 输出更统一。

## 自动化影响

本轮优化已经接入现有自动化链路：

- `daily_run` 继续支持 `--boost-keywords`
- `source_audit` 继续维护 `generated_sources.yaml`
- GitHub Actions 的每日任务和来源审计任务继续复用同一套配置
- 排序审计文件 `top_ranked_candidates.json` 继续生成

## 当前已知状态

OpenAI Help Center 的两个 release/changelog 入口在最近一次来源审计中发现不可用，已由系统自动写入 `generated_sources.yaml` 停用：

- `openai-chatgpt-release-notes`
- `openai-codex-changelog`

OpenAI Newsroom RSS 仍保持启用。由于正文页可能返回 403，本轮优化后系统会使用 RSS 摘要作为稳定降级来源。

## 后续建议

1. 优先寻找 ChatGPT 和 Codex 更稳定的官方 changelog 替代入口。
2. 继续观察 7 到 14 天的 `top_ranked_candidates.json`，按实际日报质量微调权重。
3. 对 Gemini、Kimi、GLM 的 changelog 解析继续做日期与标题清洗，避免页面级标题进入日报。
4. 如果某个产品线连续多日只有 `scanned_not_found`，优先检查其官方源是否改版，而不是引入第三方媒体填坑。
